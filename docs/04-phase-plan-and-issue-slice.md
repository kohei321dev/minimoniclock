# フェーズ計画と Issue 分割

## 現在フェーズ

- [事実] 要求定義と要件定義は、当初構想の全体像としては初版完了。
- [事実] フェーズ1の MVP は「時計常時表示」と「縦横時刻 JUSTFIT 設計」に絞る。
- [推測] 技術選定は WPF / .NET が第一候補。ただし正式採用は ADR または設計メモで確定する。
- [未検証] 実機 DPI、Windows 表示スケール、論理ピクセルサイズ、回転時挙動は未確認。

## Phase 1 MVP

目的は、JAPANNEXT `JN-MD-IPS7842` 上で、縦向き・横向きのどちらでも視認性の高い 24 時間デジタル時計を常時表示することである。

### 含める

- Windows デスクトップアプリの最小起動。
- 24 時間表記のデジタル時計。
- `400x1280` 相当の縦向き表示。
- `1280x400` 相当の横向き表示。
- ウィンドウサイズに追従する JUSTFIT レイアウト。
- ログ保存なし、外部通信なし。

### 含めない

- ポモドーロタイマー。
- カウントダウンタイマー。
- アラーム。
- 完了時 `0` 表示。
- 設定保存。
- 自動起動。
- 配布インストーラー。

## Phase 2 拡張候補

- ポモドーロタイマー。
- カウントダウン。
- アラーム。
- 完了表示。
- ローカル設定保存。
- 表示先ディスプレイ復元。
- Windows 起動時の自動起動。

## Phase 2 の優先順位

Phase 2 は「時計として常時表示できる状態」を壊さず、時間管理機能を小さく足す。最初から複数機能を同時に入れず、表示、状態管理、通知の順にリスクを分ける。

### P1: まず入れる

- ポモドーロの手動開始、一時停止、リセット。
- カウントダウン完了時の画面上の `0` 表示。
- Phase 2 用の最小 view model と単体テスト。

### P2: P1 の後に入れる

- 任意分数のカウントダウン。
- ポモドーロの作業時間と休憩時間の切り替え。
- アラーム到達時の画面表示。

### P3: 後回しにする

- アラーム音、スヌーズ、Windows 通知。
- 設定保存、表示先ディスプレイ復元、自動起動。
- Pomodoro セット数表示。

## Phase 2 最小 Issue 候補

### 1. タイマー状態モデルを追加する

目的: 時計表示から独立したタイマー状態を扱えるようにする。

含める:

- `Idle`、`Running`、`Paused`、`Completed` の状態。
- 残り時間の計算。
- 手動 tick による単体テスト。

含めない:

- WPF 画面への統合。
- 音、通知、設定保存。

完了条件:

- 日付またぎや一時停止再開で残り時間が破綻しないことを単体テストで確認できる。

### 2. ポモドーロ最小 UI を追加する

目的: 25 分の作業タイマーを画面上で開始、一時停止、リセットできるようにする。

含める:

- 既定値 25 分のポモドーロ。
- 開始、一時停止、リセット操作。
- 完了時の `0` 表示。

含めない:

- 休憩タイマーの自動開始。
- セット数表示。
- 設定保存。

完了条件:

- 縦向き `400x1280` と横向き `1280x400` 相当で、時計表示と操作ボタンが重ならない。

### 3. 任意カウントダウンを追加する

目的: ポモドーロとは別に、任意の分数でカウントダウンできるようにする。

含める:

- 1 分から 99 分までの入力。
- 開始、一時停止、リセット操作。
- 完了時の `0` 表示。

含めない:

- 秒単位入力。
- 複数タイマー同時実行。
- 履歴保存。

完了条件:

- 無効な入力でアプリが落ちず、状態が維持される。

### 4. アラーム最小 UI を追加する

目的: 指定時刻に画面上で気付ける最小アラームを追加する。

含める:

- 24 時間表記の時刻指定。
- 到達時の画面表示。
- 停止操作。

含めない:

- アラーム音。
- スヌーズ。
- 複数アラーム。

完了条件:

- 日付またぎの次回到達時刻が明確で、過去時刻を指定しても意図せず即完了しない。

### 5. Phase 2 設定保存の範囲を決める

目的: 保存する設定と保存しない履歴を分ける。

含める:

- 保存候補の一覧化。
- 保存しないログ、履歴、個人情報の境界。
- 設定ファイル破損時の扱い。

含めない:

- 設定保存の実装。
- 自動起動や表示先復元。

完了条件:

- 実装前に、保存してよい値と保存しない値が docs 上で判断できる。

## 最小 Issue 単位

1. GitHub 初期化と docs 初回 push。
2. 技術選定 ADR 作成。
3. WPF / .NET の最小アプリ scaffold。
4. 時計表示の domain / view model 実装。
5. 縦向き `400x1280` レイアウト実装。
6. 横向き `1280x400` レイアウト実装。
7. ウィンドウ常時表示・ボーダーレス表示の検討と実装。
8. 実機 DPI / スケーリング検証。
9. Phase 1 受け入れテスト整理。
10. Phase 2 候補 issue 化。
