# 実現性・設計検討

## 実現性の結論

- [事実] Windows 上でローカルデスクトップアプリとして、時計、カウントダウン、アラームを表示することは一般的な機能であり実現可能。
- [事実] JAPANNEXT 公式ページでは、`JN-MD-IPS7842` は `400x1280`、縦横使用可能、Windows のみに対応、スピーカー非搭載と説明されている。
- [事実] ウィンドウサイズに応じたレスポンシブレイアウトで、縦向き・横向きの両方に対応できる。
- [推測] `JN-MD-IPS7842` 専用の固定ピクセルに合わせるより、実際のウィンドウサイズとアスペクト比からレイアウトを決める方が、Windows のスケーリング差分に強い。
- [未検証] 対象ディスプレイの OS 認識、物理解像度、スケーリング、回転時の挙動は実機測定が必要。

## 技術候補

### 候補 A: WPF / .NET

Windows 専用アプリとして作る場合の第一候補。

- 利点: Windows デスクトップアプリとして成熟している。Viewbox やベクター描画で時計表示を拡縮しやすい。常に手前、枠なしウィンドウ、マルチディスプレイ配置、DPI 対応を扱いやすい。
- 懸念: 新しい Windows UI の見た目を標準で得たい場合は WinUI 3 に劣る可能性がある。
- 向いている条件: Windows 専用、小型画面常時表示、軽量、実装リスクを抑えたい場合。

### 候補 B: WinUI 3

Windows 専用アプリとして作る場合の有力候補。

- 利点: Windows との親和性が高い。単体アプリとして配布しやすい。常に手前、ウィンドウ制御、通知などを扱いやすい。
- 懸念: WPF より初期実装や配布まわりの調査が重くなる可能性がある。
- 向いている条件: Windows 11 らしい modern UI、Windows App SDK、MSIX 配布を重視する場合。

### 候補 C: Tauri

Web UI と Rust の軽量デスクトップラッパーを使う候補。

- 利点: HTML/CSS で細かいレイアウト調整がしやすい。Electron より軽量にしやすい。
- 懸念: Windows 固有のウィンドウ制御や通知で追加調査が必要。Rust/Tauri のビルド環境が必要。
- 向いている条件: 見た目を細かく作り込みたいが、Electron の常駐負荷を避けたい場合。

### 候補 D: Electron

Web UI で作るデスクトップアプリの標準的な候補。

- 利点: UI 実装が速い。Web 技術の資産を使いやすい。ウィンドウ制御の事例が多い。
- 懸念: 常時表示アプリとしてはメモリ使用量が大きくなりやすい。
- 向いている条件: まず素早くプロトタイプを作り、軽量性より開発速度を優先する場合。

## 推奨方針

Phase 1 MVP は、次の優先順で開始する。

1. WPF / .NET。
2. WinUI 3。
3. Tauri。
4. Electron。

現時点では、Phase 1 の要求が「Windows の小型サブディスプレイ専用」「時計常時表示」「軽量」「マルチディスプレイと DPI の扱いが重要」なので、第一候補は WPF / .NET とする。Electron はプロトタイプ用途の候補に留める。

## レイアウト方針

- 固定ピクセルに依存せず、ウィンドウの `width / height` で縦横を判定する。
- 公式解像度 `400x1280` と横向き相当 `1280x400` を代表サイズとして検証する。
- 時計表示は画面内に収まる最大サイズを計算する。
- 操作ボタンは通常時に邪魔にならない位置へまとめる。
- 小型画面では説明文を出しすぎず、アイコンまたは短いラベルを使う。
- 完了時は通常の時計表示よりも強い視覚状態に切り替える。

## 状態設計

MVP では状態をシンプルに保つ。

- `clock`: 現在時刻を表示。
- `pomodoro`: 作業タイマーの残り時間を表示。
- `countdown`: 任意またはプリセットの残り時間を表示。
- `alarm`: アラーム設定済み状態と到達状態を表示。
- `finished`: タイマーまたはアラーム完了状態。`0` または完了表示を大きく出す。

ログを保存しないため、永続化対象は必要最小限にする。設定保存を行う場合も、履歴ではなく現在設定だけに限定する。

## タイマー精度方針

- 表示更新は 1 秒単位でよい。
- タイマー残時間は「開始時刻 + duration - 現在時刻」から計算し、単純な interval 減算だけに依存しない。
- スリープ復帰後は現在時刻から残り時間を再計算する。
- 完了後は負の時間を表示せず、`0` または完了状態に丸める。

## テスト観点

- 横長サイズで時刻がはみ出さない。
- 縦長サイズで時刻がはみ出さない。
- 代表サイズを複数用意してスクリーンショットで確認する。
- カウントダウンが 0 で止まる。
- 一時停止と再開で残り時間が破綻しない。
- アラーム到達時に画面表示が切り替わる。
- ネットワークなしで動作する。
- ログファイルや履歴ファイルを生成しない。
- DPI 100%、125%、150% で文字切れしない。
- スリープ復帰後に時計とタイマー状態が破綻しない。
- サブディスプレイ切断時に画面外へ復帰不能にならない。

## GitHub 公開時の注意

- ローカルパス、ユーザー名、private URL、secret、token を含めない。
- 実機スクリーンショットを載せる場合は、個人情報や通知が写り込んでいないか確認する。
- README には、対応 OS、目的、MVP 機能、未対応事項を明記する。
- ライセンスを決める。
- 使用ライブラリ、フォント、音源のライセンスを確認する。
- secret scan を実施してから push する。
